# Security Incident — 2026-05-15

**Status:** vulnerabilities closed; assume data already exfiltrated; incident response pending.

**Authored:** 2026-05-15 (assessment + remediation conducted same day)
**Scope:** [https://your-portal.vercel.app](https://your-portal.vercel.app) and the shared Supabase project `mqikfwwbrrqkcrwrsfyg` ("takeover").

---

## TL;DR — what could have been seen and taken

Anyone on the internet — with no credentials, no auth, no insider access — could:

1. **Read the entire codebase** off the live Vercel deployment (README, migrations with RLS policy text, every Edge Function's source).
2. **Read 228 full identity records** from `account_owners` (Australian bowlers): name, DOB, address, driver licence number + card number + expiry, Medicare number + reference, passport details, bank name + balance, plaintext portal password, plaintext bet365 password.
3. **Read 5,052 `betting_accounts` rows** including bookmaker linkage, balances, status, deposit method, PIN.
4. **Read 112 `proxies` rows** including Tailscale internal IPs, device serials, ISPs, geographic distribution.
5. **Read `organization_members`** revealing the owner's Supabase user UUID and organisation UUID.
6. **Hijack any account** whose signup form wasn't yet completed — by POSTing their email to the signup endpoint and being handed back a working session token, without the password.
7. **Promote themselves** from `lead` → `confirmed` by signing up and PATCHing their own row, bypassing the entire manual-approval and payment workflow.
8. **Burn Twilio credit** by calling `send-portal-sms` with any phone number and any message body — the function had no auth check beyond the public anon key.
9. **Spam the lead funnel** unlimited times by calling `insert_portal_lead` or `signup-and-auth` with no rate limit.

Items 1–5 are **already-realised exposure**: those endpoints were anonymously readable for an unknown window, so we must assume the data is in the wild. Items 6–9 were latent vulnerabilities that could have been triggered at any point.

## Who is affected

### Bowlers (the 228 individuals in `account_owners`)

- Their full identity dossier — DL, Medicare, passport, address, DOB, mobile — is in the wild. This is **everything an identity thief needs to** open new bank accounts, file fraudulent tax returns, request replacement IDs, sign up for credit, or bypass KYC at other gambling operators.
- Their **bet365 / portal / bank passwords are exposed in plaintext**. If those passwords are reused on email, banking, or anywhere else, those accounts are also compromised.
- Their **face photo (selfie), licence front/back, Medicare card** are referenced. The image URLs are now known. The bucket access control is still intact (the bucket is private and RLS-enforced), so the actual image bytes were NOT exposed — but anyone with a leaked auth-user-id mapping to a bowler could potentially still reach them.

### Account owners (the 5,052 betting accounts)

- Wherever credentials are stored, treat them as known. Many will reuse passwords across bookmakers. The bookmakers themselves should be notified if their account-takeover risk is non-trivial.

### Your organisation

- Operational structure leaked: how many proxies, where they are, what bookmakers are in scope, how the kanban onboarding moves bowlers from lead → confirmed.
- Internal device fleet (Tailscale IPs, phone serial numbers) is now mapped.
- Owner's Supabase auth UUID is known — could be targeted for credential stuffing against Supabase Auth or any future admin endpoint.

### The Telegram community

- The invite URL was hardcoded in `scripts/write-config.js`. Anyone who scraped the repo has it. The Telegram channel may receive spam or unsolicited joiners.

### Twilio account

- The auth token was reachable by any anon JWT holder via the `send-portal-sms` edge function. Attackers could have used your Twilio number to send arbitrary SMS for the cost of your credit. There's no way to know from outside whether this was abused; check Twilio usage logs.

## What an attacker who scraped the data can do

Quick reference, ordered by ease and impact:

| Capability | Required input | Mitigation now |
|---|---|---|
| Identity theft (open credit / bank account in bowler's name) | DL, Medicare, address, DOB | None — data is out. Bowler must monitor credit. |
| Log in to bowler's bet365 | `bet365_password` from `account_owners` | Force every bowler to reset bet365 password. |

## What was changed

### Database

- New migration: [`supabase/migrations/20260515000000_lock_anon_access.sql`](../supabase/migrations/20260515000000_lock_anon_access.sql). Enables RLS on every `public.*` table, revokes all anon/public grants on tables and functions, re-grants only what the landing flow needs, revokes column-level SELECT on stored plaintext credentials, creates a `_rls_audit` view for monitoring.
- New migration: [`supabase/migrations/20260515010000_pentest_fixes.sql`](../supabase/migrations/20260515010000_pentest_fixes.sql). Drops `bowlers.encrypted_password`, adds `bowlers_block_self_promotion()` trigger, drops `insert_portal_lead` RPC, drops any `PUBLIC`-targeted policies on sensitive tables.
- New migration: [`supabase/migrations/20260515020000_signup_rate_limit.sql`](../supabase/migrations/20260515020000_signup_rate_limit.sql). Creates `signup_attempts` table.
- New migration: [`supabase/migrations/20260515030000_atomic_signup_rate_limit.sql`](../supabase/migrations/20260515030000_atomic_signup_rate_limit.sql). `try_record_signup_attempt()` function with advisory lock. Tightens the bowlers INSERT policy to forbid admin-only columns at insert.

### Edge functions

- [`signup-and-auth`](../supabase/functions/signup-and-auth/index.ts): CORS allowlist via `PORTAL_ALLOWED_ORIGINS`; removed the re-auth-by-stored-password branch; stops writing `encrypted_password`; calls `try_record_signup_attempt` for atomic rate limiting; returns 429 with `Retry-After` on rejection.
- [`send-portal-sms`](../supabase/functions/send-portal-sms/index.ts): requires `bowler_id`; loads `mobile` / `first_name` / `organization_id` from the bowler row; ignores client-supplied mobile and body; one SMS per bowler per 5 minutes; CORS allowlist.
- [`enable-portal-login`](../supabase/functions/enable-portal-login/index.ts): CORS allowlist; stops returning `credential_password`; supports admin password rotation via `auth.admin.updateUserById`.

### Frontend

- All runtime assets moved to `public/`. [`vercel.json`](../vercel.json) `outputDirectory: "public"`. README, docs, migrations, edge function source, dev scripts no longer served.
- [`vercel.json`](../vercel.json) adds `Content-Security-Policy` (without `'unsafe-inline'` on `script-src`) and `Permissions-Policy`. HSTS + X-Frame-Options + nosniff + Referrer-Policy were already present.
- [`.vercelignore`](../.vercelignore) excludes source/dev/Claude artefacts from the upload.
- [`scripts/write-config.js`](../scripts/write-config.js) fails the build if `SUPABASE_URL` / `SUPABASE_ANON_KEY` / `PORTAL_ORGANIZATION_ID` are missing. No fallback values.
- [`public/account.js`](../public/account.js) replaces `innerHTML =` string concatenation with `createElement` + property setters. Removes the stored-self-XSS sink.

### Monitoring

- New script: [`scripts/security-smoke.sh`](../scripts/security-smoke.sh). Runs against the live API + portal. Fails if any of the 17 sensitive tables returns rows to anon, or if any of the 14 source-file paths becomes accessible. Should be run on a 15-minute cron until the mystery re-grant is understood.

## Lessons (for the next architecture decision)

1. **`outputDirectory: "."` is never safe** for a project that also contains migrations and Edge Function source. Vercel/Netlify/Cloudflare Pages all default to serving everything in the deploy root. A `public/` (or `dist/`) build output is a hard requirement.
2. **The Supabase anon key is only safe if every table has RLS enabled AND the GRANT to anon has been revoked.** Enabling RLS without a policy returns empty rows for SELECT — but only if there's no permissive policy (a "TO public USING (true)" on `bookmakers` was the culprit). Defence in depth means both layers.
3. **`REVOKE SELECT (column)` is a no-op when the table-level SELECT is granted.** Column-level revocation only works when there is no broader grant. To genuinely hide a column from authenticated users, either drop the column or expose the table only via a view that omits the column.
4. **Storing plaintext credentials for re-authentication is an account-takeover vector by definition.** The signup-and-auth re-auth path was the worst single bug found. Any time a server stores a user's password in a readable form so it can "log them back in," that's a vulnerability waiting for the email column to leak.
5. **Rate limiting via "check then insert" races under concurrency.** Always either (a) use an atomic SQL function with a row-level lock or advisory lock, or (b) put the rate-limit state in a single row that you `UPDATE … RETURNING` to detect over-quota in one query.
6. **CORS is not access control.** The `Origin` header is set by browsers; curl, server-side attackers, and any non-browser client trivially bypass it. CORS protects browsers from cross-site requests carrying their cookies — that's it. Anything you need to actually deny must be denied at the auth layer.
7. **A rate-limit, a captcha, and a server-side bowler lookup are all separate defences.** SMS abuse against `send-portal-sms` was closed by (a) requiring a real bowler_id, (b) loading mobile from the row not the body, and (c) the per-bowler rate limit. Each one alone would have been bypassable; together they're a wall.
